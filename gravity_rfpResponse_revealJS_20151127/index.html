Problem Statement Part 1

Develop an Open Source Application to Manage the Inventory, Fulfillment, and Internal Tracking within a Material Handling Environment (Warehouse).

Problem Statement Part 2

This system will specifically track:

  • Orders
  • Items
  • Workers
  • Pods (shelves of items)

Assumptions

  • This system is for a single warehouse
  • Bots and Pods will be treated as a single item
  • Activites outside the warehouse "walls" will not be managed
  • New items will be readily available for replenishment

Out of Scope

  • Shopping cart or ordering system
  • Tracking Bots
  • Management of 3rd party sellers
  • Tracking the shipping of items (i.e. in transit, arrival)
  • Floorspace of warehouse (location and optimization of pods, bots, workers)

Project Management Approach

  • 15 minute scrum will kick off each Lab Session
  • Scrum will identify goals for each sprint
  • PM will assign tasks to developers to meet each set of scrum goals
  • Each sprint will be self-contained; activities should not roll over into the next sprint
  • Activity durations will be estimated before each sprint begins
  • If a Developer would like to work ahead, proposed activities need to be coordinated with the PM.

Teams

  • During the daily scrum, developers will be broken into code pairs
  • One or two developers will be floating code help / code QA
  • Code QA will be a rotating position so that all developers will have an opportunity to perform that role
  • Teams will be dynamic, both in terms of membership and tasks assigned
  • If a task needs further development and is identified during the daily scrum, the previous developer team has the option to continue work

Order Tracking Process

Order Tracking Process, v1: Steps

  1. Order is recieved at /order/new
  2. Order contains items, quantities and shipping address
  3. Order is assigned an internal tracking number and timestamp when ordered was recieved
  4. Order is assigned to a (single) Packer
  5. Worker completes order
  6. Order is verified by a (single) Quality Inspector
  7. Order is assigned shipping information and is shipped

Order Tracking Process, v1: Flow Chart

Order Tracking Process, v1: Enpoints

The following are potential endpoint information for the order process,v1

  1. Status of a particular order (recieved, assigned, packed, inspected, shipped)
  2. Assigned workers for an order (Packer, Quality)
  3. Shipping information for an order (method, tracking number)
  4. Time to complete an order

Order Tracking Process, v2: Steps

  1. Order is recieved at /order/new
  2. Order contains items, quantities and shipping address
  3. Order is assigned an internal tracking number and timestamp when ordered was recieved
  4. Order is assigned to an available Packer
  5. Worker completes order
  6. Order is verified by an available Quality Inspector
  7. If an issue is found, order is assigned an available Problem Solver
  8. Order is assigned to a Shipper; shipping information is created and is shipped

Order Tracking Process, v2: Enpoints

The following are potential endpoint information for the order process, v2:

  1. Status of a particular order (recieved, assigned, packed, inspected, problem solving, shipped)
  2. Assigned workers for an order (Packer, Quality, Problem Solver, Packer)
  3. Shipping information for an order (method, tracking number)
  4. Time to complete an order
  5. Roll-up info: amount of orders at each stage, average time to complete an order...

Item Tracking Process

Item Tracking Process, v1: Steps

  1. Items are recieved and assigned to a Pod
  2. Items are pulled from a Pod and placed into a Package to complete an order
  3. If quantity of an item falls below a threshold, a replishment order is triggered

Item Tracking Process, v1: Enpoints

The following are potential endpoint information for the item tracking process,v1

  1. Status of a particular item (roll-up: quantity on hand, number shipped over a timeframe, specific pods holding the item)
  2. List of orders containing that item

Item Tracking Process, v2: Steps

  1. Items are recieved and assigned to a Quality Assurance Worker
  2. Items that pass QA are assigned an available Pod. Items that don't are sent to a Problem Solver.
  3. Items are pulled from a Pod and placed into a Package to complete an order
  4. If quantity of an item falls below a threshold, a replishment order is triggered

Item Tracking Process, v2: Enpoints

The following are potential endpoint information for the item tracking process, v2:

  1. Status of a particular item (roll-up: quantity on hand, number shipped over a timeframe, specific pods holding the item, quantity in inspection, quantity in problem solving, number on order)
  2. List of orders containing that item

Worker Tracking Process

Worker Tracking Process

  1. Workers can be Quality Assurance (items), Problem Solvers (items), Packers, Quality Assurance (packages), Problem Solvers (packages) and Shippers
  2. Each worker will need slightly different sets of data. For example, a Packer will need to know what orders have been assigned and what items are in the order.
  3. This data will be mapped out in a future lab

Worker Tracking Process: Enpoints

The following are potential endpoint information for the worker tracking process

  1. Status of a particular worker (orders they are currently assigned, working on and completed).
  2. Roll-up of number of workers performing certain tasks and durations. This data could be used for warehouse optimization.

Worker Tracking Process v1 vs v2

V1 activities will integrate order and item tracking from V1

V2 activities will integrate order and item tracking from V2

Pod Tracking Process

Pod Tracking Process

  1. Pods have limited storage. Once a pod is filled with items, no more items can be added.
  2. Pods will need to know what items are on the pods
  3. Pods will need to know if they are (1) being loaded with items, (2) being idle, (3) at a worker's station, or (4) in maitenance

Pod Tracking Process: Enpoints

The following are potential endpoint information for the pod tracking process

  1. Status of a particular pod (items on a pod, status).
  2. Roll-up of number of pods at various stations, roll-up of pods containing a certain item. This data could be used for warehouse optimization.

Pod Tracking Process v1 vs v2

V1 activities will integrate order and item tracking from V1

V2 activities will integrate order and item tracking from V2

Pod Queque and Maitenance activities will be part of v2
